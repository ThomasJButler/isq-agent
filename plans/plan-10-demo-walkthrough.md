### Section 4 — Architecture under the hood (3 minutes)

> "Let me show you why it's structured the way it is."

[Switch to n8n canvas view]

> "Tier 1 is n8n. It owns the workflow, the file I/O, the rendering. The brief explicitly said 'not entirely code-based' so n8n is the orchestrator."

[Switch to VSCode, open `rag-service/app/main.py`]

> "Tier 2 is a small Python FastAPI service. It owns the RAG core — embedding via Voyage, vector search via Pinecone, generation via Claude. They talk over HTTP. The integration is the design — I considered building everything in n8n's code nodes, but I'd already solved enterprise RAG in Morpheus, and rebuilding it inside n8n's runtime would be lower quality than what I already have. So I split: n8n where it shines, Python where I have existing expertise."

[Open `rag-service/app/rag/generator.py`]

> "The answer generator does one Claude call per question, using tool-use to force JSON output. The system prompt enforces strict rules — use only the provided chunks, prefer policies over historical ISQs, never invent, self-score across four dimensions. I borrowed the strict-rules pattern from SQL-Ball, and the single-call-multi-field shape from NewsPerspective."

[Open `app/confidence/aggregator.py`]

> "Confidence is hybrid — the LLM's four-dimensional self-score combined with retrieval similarity as a sanity check. The pattern comes from ReviewBot's multi-dimension code analysis. If the LLM over-claims relative to retrieval, the sanity check downgrades."

**Timing:** ~3 minutes. This is the architecture story + portfolio-convergence proof.

## 4. Likely Q&A (with your answers)

**"Why two tiers instead of all in n8n?"**

> "Two reasons. First, I'd already built a production RAG core in Morpheus — query rewriter, retrieval, reranking. Rebuilding it inside n8n's code nodes would have been lower quality than what I had. Second, the brief said 'not entirely code-based.' Two-tier honours that — n8n is genuinely the orchestrator, not a thin HTTP relay. Code earns its place where it does what n8n can't do well."

**"Why Voyage instead of OpenAI embeddings?"**

> "Anthropic-aligned and I'm investing in the Anthropic ecosystem long-term. Quality difference vs OpenAI text-embedding-3-small is marginal for this corpus. Both 1024 dims. Functionally interchangeable. For the walkthrough story, Voyage signals intent to stay in the Claude partner ecosystem."

**"What happens if Claude is down?"**

> "Per-question fail-fast with retries. n8n retries 3× with exponential backoff. If all retries fail for one question, that question's `confidence` is null and `needs_review: true` with reason 'generation failed.' Other questions still complete. Partial completion is a first-class design decision, not a regression."

**"How do you stop the LLM hallucinating citations?"**

> "Two layers. First, the system prompt is strict — 'use only the chunks provided.' Second, every claimed citation is verified against the chunks we actually gave the LLM. If it cites a source_id we didn't provide, we log a warning and downgrade the cites_policy score by 0.2. Rare in practice but caught when it happens."

**"How would you scale this if Northstar got 100 ISQs a day?"**

> "Two things change. First, horizontal scale on the RAG service — it's stateless, so `docker compose up --scale rag-service=5` would give 5× throughput. Second, async processing — current design is synchronous within one ISQ. For batch ingestion I'd add a queue (Redis or Postgres) and worker pattern. n8n supports both natively."

**"How did you choose the confidence weights?"**

> "Documented in the aggregator module's docstring. cites_policy at 0.40 because ungrounded is the worst failure mode for audit-facing tools. complete at 0.15 because partial-but-correct is recoverable; complete-but-wrong is not. The exact numbers are publicly visible — Lee can audit them in the code. I'd treat them as v1 — tune against more eval data over time."

**"What's the cost per ISQ?"**

> "About 8 cents for a 20-question questionnaire. Voyage embedding is negligible. Claude generation is the bulk — Sonnet at roughly $0.004 per question. The cost is in the response metadata of every run, not estimated."

**"Why TDD?"**

> "Two reasons. The walkthrough card: I built this TDD. Every module has tests written before implementation. Lee, you can run pytest and see them pass. The engineering reason: TDD codebases survive maintenance. If I'm not the one supporting this in six months, the test suite tells whoever picks it up what each component is supposed to do."

**"Could this be a product?"**

> "Yes — that's actually why I scoped it the way I did. The architectural pattern (knowledge-grounded generation, voice constraint, multi-format output) is reusable for any vendor questionnaire workflow. SaaS companies that field a lot of supplier security questionnaires would be the obvious market. I'd build out multi-tenant isolation, Notion/Slack integrations, and a self-service knowledge upload UI."

**"What's the biggest weakness?"**

> "Two honest ones. First, the PDF overlay isn't true overlay — I typeset rather than write on top of the original. Compliance teams who need the literal form filled in would notice. Second, the system prompt + few-shot is v1 — with more eval cycles I'd tune both significantly. I lay out the eval discipline in the README but didn't get to ship v2 of the prompt within the assessment window."

## 6. Traps to AVOID

1. **Don't apologise.** Avoid "this is just a v1" / "I didn't have time to" before showing things. Show first, contextualise after.
2. **Don't oversell.** "This is production-ready" → eye-roll. "This is the shape of how I'd build production" → respect.
3. **Don't go too deep on Morpheus.** Lee saw it Monday — don't recap. One sentence is enough.
4. **Don't read your own code aloud.** Talk to the design decisions, not the syntax.
5. **Don't argue with feedback.** If Lee says "I'd have done X instead" — "great call, I considered X and went Y because Z" is the senior response.
6. **Don't ramble in Q&A.** Answer the question asked, then stop. If they want more they'll ask.
7. **Don't forget to ASK questions back.** "What's a typical questionnaire size at RiverAI?" / "What does your current ISQ workflow look like?" — shows engagement.
