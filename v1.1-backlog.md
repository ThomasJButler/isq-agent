# v1.1 Backlog

What I'd build next given more time. Every item here is a deliberate scope call, not something I forgot.

## Shipped in v1.1

The record of what got built in v1.1 (not deferred):

- **Hosted deployment.** Dashboard on Vercel, backend on Render (Docker), CORS env-driven. Free to use, no per-user key, for a limited time.
- **Cost protection.** Per-IP rate limiting (slowapi), a question cap on `/process-questionnaire`, and a request-body size cap enforced at the ASGI layer (`MaxBodySizeMiddleware`) before the multipart parser buffers an upload, in front of Anthropic workspace rate limits + a monthly spend cap. See `SECURITY.md` §5.
- **Run store + live dashboard.** `run_id` + `GET /runs/{id}`; the results screen fetches the real run (no more build-time mock).
- **File upload ingest.** `POST /runs` accepts an uploaded blank ISQ (PDF / DOCX / XLSX), extracts the text and questions server-side (reusing `document_processor` + `QuestionExtractor`), answers and stores the run, and returns the canonical envelope for the dashboard to navigate to. Blocking work runs off the event loop via `asyncio.to_thread`.
- **Downloads wired** to `POST /render` (DOCX / XLSX / JSON).
- **Settings:** server-managed keys (no key inputs), plus Claude Sonnet 4.6 and Opus 4.7 in the model picker.
- **Mobile + readability pass:** no horizontal scroll, right-sized text and confidence readouts across every route.

## Shipped in v1.2

What got built in v1.2, on top of v1.1:

- **~6× faster answering.** Client singletons + startup pre-warm, bounded concurrency over the per-question loop, and query rewriting moved to Haiku took a 20-question run from **~4 min to ~40 s** and silenced ~100 lines of Pinecone plugin-discovery noise per run. Full write-up in [`docs/performance-optimisation.md`](docs/performance-optimisation.md).
- **Model picker wired end to end.** The Settings picker was cosmetic; it now threads the chosen model through `createRun`/`uploadRun` → `/runs` → the generator, validated against an allowlist. **Claude Opus 4.8** added alongside Sonnet 4.5/4.6, Opus 4.7 and Haiku 4.5. Query rewriting stays on Haiku.
- **Processing showcase.** The submit state now renders an honest pipeline view (timeline + stat tiles + generic activity log) driven by a real elapsed clock, replacing the bare spinner.
- **n8n orchestration scaffold.** An importable workflow ([`n8n/`](n8n/)): Form Trigger → `POST /runs` → render ×3 → deliver, verified to import cleanly into n8n 2.22.5. Moved off "designed, not shipped" (below) to "importable scaffold, finished in the editor."
- **Realistic examples.** The three canned example questionnaires are now full 20-question fictional suppliers (the "10 not 20" fix), replacing fabricated 10/7/5-question stubs.
- **Assessment docs.** `technical-walkthrough.md` (the 7 explain-points), `requirements-coverage.md` (every brief criterion → evidence), `unplanned-fixes.md` (the work the plan didn't predict).

## Rendering

- **Typeset PDF renderer (reportlab).** Deferred from v1 on purpose: DOCX, XLSX and JSON cover the formats suppliers actually send back, so a properly typeset PDF was the right thing to push out rather than rush.

## Retrieval quality

- **Citation source_id to human-readable policy reference mapping.** Right now a citation points at a chunk; I'd map that back to a named policy and section so a reviewer can find the source in seconds instead of grepping.
- **A reranker pass on retrieval results.** The current top-k plus source weighting is solid, but a dedicated reranker would tighten the ordering on the harder, more ambiguous questions where the right chunk sits at position three or four.
- **Extra Voyage embedding models (the Plan 13 spike).** I want to benchmark alternative embedding models against our corpus before committing, because the gains are worth measuring rather than assuming.

## Answer generation

- **Extended-thinking second pass for flagged answers.** When confidence drops below threshold, I'd run a second pass with extended thinking so the low-confidence cases get the extra compute they deserve, and the easy ones stay fast and cheap.
- **More few-shot examples in the answer prompt.** A wider, curated set of worked examples would sharpen tone and structure on the question types we see least often.

## Orchestration and product

- **Run the n8n workflow end to end + add more triggers.** The workflow is built and imports cleanly (v1.2, `n8n/`); the remaining increment is finishing it in the editor, running a paid end-to-end submission, and adding the email / webhook / cloud-folder triggers beyond the manual form upload.
- **Per-user authentication.** The hosted demo is deliberately open (fenced by cost controls, not a login). A real deployment handling customer questionnaires would need auth + per-tenant isolation of runs.

## Evaluation

- **A self-evaluation / eval harness with ground-truth scoring.** Score answers against a known-good set so I can measure regressions properly instead of eyeballing them, and tune retrieval and prompts against real numbers.

## Dependencies and security

- **Frontend postcss advisory (done in v1.1).** Dependabot flagged `postcss <8.5.10` (GHSA-qx2v-qp2m-jg93), pulled in transitively by `next@16.2.6`. Resolved with an npm `overrides` forcing `^8.5.10`; the whole tree now resolves to 8.5.15 and `npm audit` is clean.
- **Bump the vulnerable pinned backend dependencies.** pip-audit flags 27 advisories, almost all in pypdf 4.3.1 (22, fixed in the 6.x line), plus langchain-text-splitters (1.1.2), starlette (transitive via FastAPI), and pytest (dev-only). Each is a major-version bump that crosses the current pins, so they want doing together with a full test re-run rather than a rushed change before the submission cut. Logged in `SECURITY.md` too.
