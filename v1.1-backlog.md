# v1.1 Backlog

What I'd build next given more time. Every item here is a deliberate scope call, not something I forgot.

## Rendering

- **Typeset PDF renderer (reportlab).** Deferred from v1 on purpose: DOCX, XLSX and JSON cover the formats suppliers actually send back, so a properly typeset PDF was the right thing to push out rather than rush.

## Retrieval quality

- **Citation source_id to human-readable policy reference mapping.** Right now a citation points at a chunk; I'd map that back to a named policy and section so a reviewer can find the source in seconds instead of grepping.
- **A reranker pass on retrieval results.** The current top-k plus source weighting is solid, but a dedicated reranker would tighten the ordering on the harder, more ambiguous questions where the right chunk sits at position three or four.
- **Extra Voyage embedding models (the Plan 13 spike).** I want to benchmark alternative embedding models against our corpus before committing, because the gains are worth measuring rather than assuming.

## Answer generation

- **Extended-thinking second pass for flagged answers.** When confidence drops below threshold, I'd run a second pass with extended thinking so the low-confidence cases get the extra compute they deserve, and the easy ones stay fast and cheap.
- **More few-shot examples in the answer prompt.** A wider, curated set of worked examples would sharpen tone and structure on the question types we see least often.

## Orchestration and product

- **Build and export the n8n orchestration workflow.** It's designed but not shipped: the RAG service does the hard work today, and wiring the full n8n flow is the next clean increment once the API surface settled.
- **Next.js dashboard glue.** Backend `run_id` plus `GET /runs/{id}`, then wire the adapter that's already tested. The frontend and the contract exist; this is the join that makes them one thing.
- **Hosted deployment.** Local Docker stack runs the whole system today. Getting it hosted is a deployment task, not a code one, so it sat below the features that prove the idea.

## Evaluation

- **A self-evaluation / eval harness with ground-truth scoring.** Score answers against a known-good set so I can measure regressions properly instead of eyeballing them, and tune retrieval and prompts against real numbers.

## Dependencies and security

- **Bump the vulnerable pinned dependencies.** pip-audit flags 27 advisories, almost all in pypdf 4.3.1 (22, fixed in the 6.x line), plus langchain-text-splitters (1.1.2), starlette (transitive via FastAPI), and pytest (dev-only). Each is a major-version bump that crosses the current pins, so they want doing together with a full test re-run rather than a rushed change before the submission cut. Logged in `SECURITY.md` too.
